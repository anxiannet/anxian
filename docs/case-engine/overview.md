# Case Engine Overview｜案件引擎总览

## 1. 定位

Case Engine 是《暗线》的案件运行与生成中枢。

它不是传统 TRPG 规则系统，也不是长篇小说生成器，而是负责把一个都市异常事件拆成可游玩的调查流程：

- 玩家收到异常信息
- 进入现场
- 做出选择
- 获得线索
- 发现矛盾
- 推理判断
- 得出真相
- 写入档案
- 触发后续事件

Case Engine 的核心目标是让玩家在极短时间内进入案件，并在一次轻量调查中获得完整体验。

原则：

- 1 分钟进入事件
- 5 分钟卷入谜团
- 10 分钟完成一次调查体验
- 单人即可完整游玩
- 所有内容都能结构化归档
- 所有真相都必须由线索支撑

---

## 2. 案件体验目标

案件必须像短剧一样直接。

每个案件开局要快速回答四个问题：

1. 发生了什么？
2. 哪里不对劲？
3. 为什么必须继续查？
4. 如果不查，可能会有什么后果？

玩家不应该先阅读大量世界观。

玩家应该先被事件抓住，然后在调查过程中逐步理解白港市、都市传说和暗线网络。

---

## 3. 核心循环

```text
异常信息
  ↓
案件开场
  ↓
场景调查
  ↓
线索获取
  ↓
矛盾发现
  ↓
推理选择
  ↓
结局判定
  ↓
档案归档
  ↓
新事件触发
```

每 1 到 3 次玩家选择，系统都应该给出一次明确反馈。

反馈可以是：

- 新线索
- 新 NPC
- 新地点
- 新证据
- 新矛盾
- 风险上升
- 档案更新
- 异常现象
- 暗线节点闪现

禁止出现大量“无事发生”的空选择。

---

## 4. Case Engine 负责什么

Case Engine 负责以下能力：

1. 案件生成
2. 场景推进
3. 选择与后果判定
4. 线索管理
5. 证据管理
6. NPC 状态管理
7. 地点状态管理
8. 风险与隐藏状态处理
9. 推理节点判定
10. 结局分支判定
11. 档案写入
12. 后续案件触发

其中档案系统已经在 `13-archive-system.md` 中单独定义。

本 overview 用于说明 Case Engine 与档案、线索、证据、场景、NPC、排行榜等系统之间的关系。

---

## 5. 核心对象

### 5.1 Case｜案件

案件是一次完整调查体验的容器。

案件包含：

- 案件编号
- 案件名称
- 一句话钩子
- 案件类型
- 推荐时长
- 风险等级
- 主要地点
- 主要 NPC
- 核心异常
- 表层真相
- 深层真相
- 关键反转
- 结局列表

示例：

```json
{
  "case_id": "CASE_001",
  "case_name": "不存在的240号房",
  "hook": "你收到一张照片，照片里你站在一扇现实中不存在的房门前。",
  "case_type": "urban_legend",
  "estimated_minutes": 10,
  "risk_level": 2,
  "main_location_ids": ["LOC_001"],
  "main_npc_ids": ["NPC_001"],
  "core_anomaly": "24楼出现系统中不存在的房间",
  "surface_truth": "有人伪造了门牌和门禁记录",
  "deep_truth": "这间房只在特定档案被读取后出现",
  "twist": "最早登记进入240号房的人，是玩家自己的旧身份"
}
```

---

### 5.2 Scene｜场景

场景是玩家可以行动的调查节点。

每个案件建议包含 5 到 8 个场景。

每个场景必须包含：

- 场景编号
- 场景名称
- 场景地点
- 第一视角描述
- 当前目标
- 可选行动
- 可发现线索
- 可能触发事件
- 场景图片提示词

场景描述默认使用第二人称第一视角体验。

不要写：

> 玩家来到公寓走廊。

要写：

> 你站在24楼走廊尽头，声控灯迟迟没有亮起。手机屏幕上的门牌号是240，但你面前只有239和241。

---

### 5.3 Choice｜选择

选择是玩家推进案件的主要输入。

每个关键场景建议提供 3 个选择：

1. 稳妥调查：风险低，获得基础线索。
2. 冒险深入：风险高，可能获得隐藏线索。
3. 社交试探：影响 NPC 信任，可能获得证词。

每个选择都必须产生后果。

后果包括：

- 获得线索
- 错过线索
- NPC 态度变化
- 风险上升
- 理智下降
- 解锁新地点
- 触发异常事件
- 写入调查日志

---

### 5.4 Clue｜线索

线索是推理的基础。

每个案件至少 8 条线索。

线索类型：

- 表层线索：容易获得，推动玩家继续查。
- 矛盾线索：与 NPC 证词或已知事实冲突。
- 隐藏线索：需要正确选择或组合线索才能发现。
- 异常线索：无法用常识解释。
- 反转线索：后期重新解释前面信息。

结构示例：

```json
{
  "clue_id": "CLUE_001",
  "case_id": "CASE_001",
  "name": "门禁记录缺页",
  "type": "contradiction",
  "scene_id": "SCENE_002",
  "acquisition_method": "查看保安室门禁导出表",
  "surface_meaning": "240号房没有登记记录",
  "real_meaning": "记录被人为删改过",
  "is_key_clue": true,
  "related_evidence_ids": ["EVD_001"]
}
```

规则：

- 真相不能没有线索支撑。
- 关键反转必须能被前文线索回收。
- 线索不能一次性解释全部真相。
- 隐藏线索必须有合理获取路径。

---

### 5.5 Evidence｜证据

证据是可被玩家查看、听见、对比或归档的内容。

证据类型：

- 图片证据
- 语音证据
- 聊天记录
- 监控截图
- 门禁记录
- 新闻片段
- 档案页面
- 物品照片

证据不是装饰。

每份证据都应至少承担一种功能：

- 推动调查
- 暴露矛盾
- 指向 NPC
- 解锁地点
- 支撑推理
- 触发档案更新
- 暗示暗线节点

---

### 5.6 NPC｜人物与账号

NPC 可以是真人角色，也可以是匿名账号、举报人、目击者、管理员、失踪者、档案代号。

NPC 需要记录：

- 显示名称
- 表面身份
- 说话风格
- 初始态度
- 隐藏动机
- 可信度
- 与案件关系
- 关键台词
- 相关线索
- 相关证据

NPC 台词必须短，像真实聊天。

不要让 NPC 长篇解释设定。

---

### 5.7 Location｜地点

地点必须全部虚构。

禁止使用真实人物、真实组织、真实公寓、真实住址、真实学校、真实物业机构作为案件主体。

地点需要记录：

- 地点编号
- 地点名称
- 所属区域
- 地点类型
- 第一视角描述
- 可调查场景
- 关联案件
- 关联 NPC
- 已发现异常
- 进入条件

地点不仅是背景，也可以作为案件触发器。

---

### 5.8 Reasoning Node｜推理节点

推理节点用于判断玩家是否已经具备得出结论的条件。

推理不是让系统直接告诉玩家答案，而是让玩家在已有线索中做选择。

结构示例：

```json
{
  "reasoning_node_id": "RN_001",
  "case_id": "CASE_001",
  "question": "240号房为什么不在系统里？",
  "required_clue_ids": ["CLUE_001", "CLUE_003", "CLUE_006"],
  "options": [
    {
      "option_id": "A",
      "text": "门牌被临时伪造",
      "result": "partial_truth"
    },
    {
      "option_id": "B",
      "text": "系统记录被篡改，房间出现条件与档案读取有关",
      "result": "deep_truth"
    }
  ]
}
```

规则：

- 推理选项必须来自已有线索。
- 错误推理可以进入失败或普通结局。
- 深层推理需要关键线索组合。
- 隐藏结局需要额外线索或隐藏档案。

---

## 6. 档案系统关系

档案系统负责沉淀玩家调查过程。

Case Engine 每次推进案件时，都可能写入档案：

- 案件档案
- NPC 档案
- 地点档案
- 都市传说档案
- 暗线档案
- 调查员日志
- 隐藏档案
- 档案关联图

写入原则：

- 档案必须来自调查。
- 档案必须可追溯来源。
- 档案必须服务推理。
- 档案不能一次性暴露真相。

Case Engine 只负责产生和更新档案事件。

档案展示、搜索、完成度和关联图规则见 `13-archive-system.md`。

---

## 7. 结局系统

每个案件至少 3 个结局。

建议结局类型：

- 普通结局：解决表层问题，但未触及暗线。
- 深入结局：发现深层真相，但付出代价。
- 失败结局：误判或逃避，异常扩散。
- 隐藏结局：组合关键线索，发现暗线网络相关信息。

结局判定依据：

- 关键线索数量
- 是否发现矛盾线索
- 是否完成推理节点
- NPC 信任状态
- 风险等级
- 理智状态
- 是否解锁隐藏档案
- 是否触发暗线节点

结局要短，但必须有冲击力。

---

## 8. 隐藏状态

Case Engine 可以维护少量隐藏状态。

建议包括：

- `sanity`：理智状态
- `stamina`：体力状态
- `risk_level`：风险等级
- `npc_trust`：NPC 信任度
- `clue_completion`：线索完整度
- `archive_completion`：档案完成度
- `darkline_exposure`：暗线接触度

隐藏状态不应变成复杂数值游戏。

对玩家展示时应转换为叙事反馈。

例如：

- 你开始头痛。
- 你觉得这个人没有说实话。
- 你的档案出现了一行你没写过的字。
- 电梯监控里，你比实际时间早到了三分钟。

---

## 9. 图片与语音生成接口

每个关键场景都应能生成图片提示词。

图片提示词必须包含：

- 第一视角或近似第一视角
- 虚构地点
- 时间与光线
- 关键物体或线索
- 悬疑氛围
- 现代都市真实感
- 不出现过度血腥
- 不出现真实品牌
- 不依赖大量文字阅读

语音证据必须短、怪、带钩子。

适合的语音证据：

- 断续电话录音
- 聊天语音转写
- 监控拾音
- 失踪者留言
- NPC 语气异常的语音片段

---

## 10. 案件生成质量检查

生成或导入案件时，Case Engine 应检查：

- 是否 30 秒内进入冲突？
- 是否适合单人游玩？
- 是否第一视角？
- 是否避免真实地名和真实机构？
- 是否有强悬念？
- 是否有反差和反转？
- 是否至少包含 5 到 8 个场景？
- 是否每个关键场景都有图片提示词？
- 是否每个关键选择都有后果？
- 是否至少包含 8 条线索？
- 是否包含矛盾线索和反转线索？
- 是否至少包含 3 个结局？
- 是否可以写入档案系统？
- 是否可以拆入数据库？

不满足时，应自动退回修改。

## 11. 推荐模块拆分

后续代码实现可按以下模块拆分：

```text
case-engine/
  case-generator
  scene-runner
  choice-resolver
  clue-manager
  evidence-manager
  npc-state-manager
  location-state-manager
  reasoning-engine
  ending-resolver
  archive-writer
  trigger-engine
```

模块说明：

- `case-generator`：生成案件骨架、场景、NPC、线索、结局。
- `scene-runner`：根据当前场景输出描述、目标和可选行动。
- `choice-resolver`：处理玩家选择与后果。
- `clue-manager`：管理线索获取、隐藏线索和线索组合。
- `evidence-manager`：管理图片、语音、聊天记录和证据元数据。
- `npc-state-manager`：维护 NPC 态度、信任度和证词变化。
- `location-state-manager`：维护地点开放状态和异常状态。
- `reasoning-engine`：处理推理节点、线索组合和真相判断。
- `ending-resolver`：根据调查结果进入不同结局。
- `archive-writer`：把调查结果写入档案系统。
- `trigger-engine`：根据档案、地点、时间、线索组合触发新事件。

## 12. MVP 优先级

第一阶段优先实现：

1. 案件数据结构
2. 场景数据结构
3. 选择与后果系统
4. 线索系统
5. 证据系统
6. 档案写入
7. 简单结局判定
8. 新手微型案件

第二阶段实现：

1. NPC 信任变化
2. 地点解锁
3. 推理节点
4. 隐藏线索组合
5. 档案关联图
6. 图片证据生成
7. 语音证据生成

第三阶段实现：

1. 暗线节点网络
2. 全服案件种子
3. 玩家痕迹
4. 排行榜
5. 情报市场
6. 案件共创
